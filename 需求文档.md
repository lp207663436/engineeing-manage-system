# 智能化工程管理系统 需求文档

## 一、项目概述

### 1.1 项目背景
本系统面向智能化工程行业,旨在为工程公司提供一套覆盖项目全生命周期的信息化管理工具,解决合同分散、报价难追溯、进度不透明、维保脱节等管理痛点。

### 1.2 系统目标
- 实现项目合同、报价、进度、维保的统一数字化管理
- 建立项目全过程档案,实现数据可追溯
- 提升跨部门协作效率,降低人工管理成本
- 为经营决策提供数据支撑

### 1.3 适用对象
项目经理、商务人员、工程实施人员、维保人员、管理层

### 1.4 系统使用范围约束 ⭐
- **本系统仅内部使用,客户不会登录系统**
- 客户相关动作(合同确认、验收签字、付款承诺等)均通过线下完成,由内部人员上传附件留痕
- 系统不提供客户账号、客户端、客户门户

---

## 二、用户角色与权限

> 权限模型:RBAC + 数据权限(无租户隔离,单租户架构)
> - **功能权限**:菜单/按钮级,按角色配置
> - **数据权限**:本人 / 本部门 / 本部门及下属 / 全部,按模块可配
> - **字段权限**(可选):合同金额等敏感字段可按角色脱敏

| 角色 | 主要职责 | 权限范围 |
|------|---------|---------|
| 系统管理员 | 系统配置、用户管理 | 全部模块读写、权限分配 |
| 管理层 | 经营数据查看、审批 | 项目总览、合同审批、报表查看 |
| 商务专员 | 合同与报价管理 | 合同录入/上传、报价上传 |
| 项目经理 | 进度管理、协调 | 项目进度维护、节点更新 |
| 实施人员 | 现场进度上报 | 进度填写、施工日志 |
| 维保人员 | 售后维保执行 | 维保任务处理、维保记录 |

## 三、功能模块详细需求

### 模块一:项目合同管理

#### 3.1.1 合同录入与建档
- 录入合同基本信息:合同编号、合同名称、甲方/乙方信息、签订日期、合同金额、工期、付款方式
- 合同分类管理(总包合同、分包合同、采购合同等)
- 合同电子件上传(扫描件 PDF/图片),支持多附件
- 合同与项目自动关联,一个项目可关联多份合同

#### 3.1.2 合同审批
- 支持自定义审批流程(金额分级、角色分级)
- 审批节点:起草 → 部门审核 → 财务审核 → 管理层审批 → 归档
- 审批意见记录、可追溯

#### 3.1.3 合同收付款管理
- 合同应收/应付计划录入
- 实际收付款记录,支持票据上传
- 收付款进度看板(已收/未收/逾期)

#### 3.1.4 合同变更与归档
- 合同变更记录(补充协议上传、变更说明)
- 合同到期/完工后归档
- 合同检索:按编号、名称、甲方、签订日期、状态查询

---

### 模块二:报价上传存储

#### 3.2.1 报价文件上传
- 支持上传报价单(Excel/PDF/图片),单文件或批量上传
- 关联对应项目或潜在项目
- 录入报价摘要信息:报价编号、报价金额、报价日期、报价人、客户名称、有效期

#### 3.2.2 报价版本管理
- 同一项目支持多版本报价(初次报价、调整报价、最终报价)
- 版本对比,标记差异
- 历史报价归档保留

#### 3.2.3 报价查询与复用
- 按项目、客户、金额区间、日期查询
- 历史报价模板复用,快速生成新报价
- 报价状态管理(草稿、已发送、已确认、已失效)

#### 3.2.4 报价与合同联动
- 中标报价可直接转换为合同基础信息
- 报价金额与合同金额一致性校验提示

---

### 模块三:项目进度管理

#### 3.3.1 项目立项与基础信息
- 项目立项:项目编号、项目名称、所属合同、项目经理、开工/竣工日期、项目地址
- 项目分阶段划分(如:设计、设备采购、施工安装、调试、验收)
- 项目团队人员配置

#### 3.3.2 进度计划编制
- 制定项目总体进度计划(甘特图展示)
- 设置关键节点与里程碑
- 各节点负责人、计划开始/结束时间

#### 3.3.3 进度上报与跟踪
- 实施人员按节点填报实际进度(百分比 + 文字说明 + 现场照片)
- 系统自动对比计划与实际进度,超期预警
- 进度变更记录留痕

#### 3.3.4 进度看板与提醒
- 项目进度总览看板(进行中/延期/完工)
- 关键节点到期提醒(站内消息、邮件)
- 项目问题/风险登记与跟进

#### 3.3.5 项目验收
- 验收资料上传(验收单、竣工资料)
- 验收状态跟踪(待验收、验收中、已验收、整改中)

### 模块四:项目维保

#### 3.4.1 设备台账管理 ⭐基础能力
- 设备档案录入:设备编号、设备名称、品牌、型号、序列号、规格参数、投运日期、保修到期日
- 设备分类管理(按系统分类:如门禁、监控、网络、楼控等)
- 设备与项目/点位关联:一个设备归属一个项目,维护型项目下可关联到具体点位
- 设备状态管理:正常 / 故障 / 维修中 / 报废
- 设备附件上传:说明书、合格证、保修卡
- 设备检索:按编号、名称、型号、项目、状态查询
- 设备维保历史:查看该设备的所有巡检记录、故障记录、维修记录

#### 3.4.2 维保合同管理
- 维保合同录入(维保期、维保范围、响应时效、维保费用)
- 与原项目合同关联
- 维保到期自动提醒
- 维保合同可关联覆盖的设备清单

#### 3.4.3 维保任务管理
- 系统按维保计划自动生成定期巡检任务(可按设备生成)
- 故障报修工单:内部提交报修(关联设备) → 派单 → 处理 → 完工反馈
- 工单状态流转:待派单、已派单、处理中、待验收、已完成、已关闭
- 工单关联设备,记录故障设备、故障现象、处理方法、更换配件

#### 3.4.4 维保记录与归档
- 维保人员上传维保记录(处理过程、更换配件、现场照片)
- 维保耗材/配件使用记录,关联设备
- 维保历史档案,支持按项目/设备/点位查询

#### 3.4.5 维保统计与分析
- 故障类型统计、响应时长分析
- 设备故障频次排行(基于设备台账)
- 维保人员工作量统计
- 设备健康度评分(基于故障次数、使用年限)

### 模块五:维护型项目(多点位)管理 ★重点

> 适用场景:对已交付项目/系统进行持续维护,一个维护项目包含多个维护点位,每个点位独立报价、独立施工、独立验收。维护费按季度结算,报价费验收后结算。

#### 3.5.1 业务模型

```
维护型项目(主合同:周期维保服务)
   ├── 维护费:按季度结算(周期性)
   │
   └── 多个维护点位 / 分项(每个点位独立报价、独立施工、独立验收)
         ├── 点位A 报价 → 施工 → 验收 → 结算(验收后结算)
         ├── 点位B 报价 → 施工 → 验收 → 结算
         └── 点位C 报价 → 施工 → 验收 → 结算
```

核心要点:**一个维护型项目 = 1 份维保主合同(季度结算) + N 个点位报价单(验收后结算)**

#### 3.5.2 维护型项目立项
- 项目类型选择「维护型项目」,区别于新建工程
- 录入项目基本信息:项目编号、项目名称、客户、维保地址、维保起止日期、项目经理
- 关联原交付项目(可选,便于追溯历史)
- 配置维保服务范围、响应时效
- **合同期校验**:维保主合同的合同期必须为 3 个月的整数倍且不少于半年(`合同期月数 % 3 == 0 且 ≥ 6`),否则保存失败并提示

#### 3.5.3 维护点位管理(核心)
- 一个维护项目下可创建多个维护点位/分项
- 每个点位录入:点位编号、点位名称、位置描述、设备/系统清单、负责人
- 点位状态:待报价 → 已报价 → 施工中 → 待验收 → 已验收 → 已结算

#### 3.5.4 点位单独报价
- 每个点位独立发起报价,生成独立报价单
- 报价内容:报价金额、报价明细(人工/材料/设备)、报价日期、有效期
- 报价审批流程(沿用合同/报价模块审批)
- 报价状态:草稿、已提交、已审批、已确认、已作废
- 报价确认后,转为该点位的施工任务

#### 3.5.5 维护费季度结算(周期性)

> 详细规则见 [季度结算规则设计规格](docs/superpowers/specs/2026-06-26-季度结算规则-design.md)

**(1) 合同期约束**
- 合同期单位为 3 个月(一个季度),最短半年
- 允许长度:3 个月的任意整数倍(半年 / 9 个月 / 1 年 / 15 个月 / 2 年 ...)
- 系统校验:`合同期月数 % 3 == 0 且 ≥ 6`,否则保存失败并提示

**(2) 季度口径(合同季度)**
- 起算点:合同生效日
- 第 N 期开始日 = 生效日 + (N-1) × 3 个月
- 第 N 期结束日 = 生效日 + N × 3 个月 - 1 天
- 合同到期日 = 生效日 + 合同期 - 1 天

**(3) 结算单生成**
- 时点:每期结束日的次日 00:00 自动生成草稿单(定时任务)
- 内容:本期维保费金额、本期巡检次数、本期维保工单汇总、关联维保记录
- 状态流转:草稿 → 审核 → 确认 → 开票 → 回款 → 关闭
- 生成时站内消息通知项目经理和财务

**(4) 金额计算**
- 单期金额 = 合同总额 ÷ 期数
- 尾差处理:前 N-1 期保留 2 位小数,最后一期 = 合同总额 - 前 N-1 期之和(吸纳尾差)
- 合同期为 3 个月整数倍,不存在按天折算场景

**(5) 金额变更**
- 触发:合同变更流程(补充协议上传 + 审批通过)
- 生效时点:从下一期开始按新金额结算,当期仍按旧金额,不按天补差
- 留痕:结算单显示"适用金额版本",变更历史可追溯

**(6) 提前终止**
- 系统不提供自动解约结算逻辑,由商务线下协商
- 通过"手动调整结算单"功能处理(需财务审批 + 填写调整原因 + 留痕审计)

#### 3.5.6 报价费验收后结算(单次)
- 点位施工完成后发起验收
- 验收通过后,该点位报价金额进入结算队列
- 生成点位结算单:点位报价金额 + 验收单 + 实际完成情况
- 结算单流转:验收通过 → 结算确认 → 开票 → 回款
- 强校验:未验收的点位不可结算

#### 3.5.7 双线结算看板
- 维护型项目结算总览,分两条线展示:
  - 维保费结算线:季度周期、应收、已收、待回款
  - 报价费结算线:按点位维度,已验收待结算、已结算待开票、已开票待回款
- 项目累计已收金额 = 维保费累计回款 + 报价费累计回款

#### 3.5.8 点位进度与验收

> 详细规则见 [验收规则设计规格](docs/superpowers/specs/2026-06-26-验收规则-design.md)

**(1) 验收人**
- 项目经理单签即生效,无需客户或第三方签字
- 客户确认通过线下获取,内部上传附件留痕(选填)

**(2) 发起时点**
- 点位施工状态置为"已完成"时,系统自动生成验收单草稿
- 项目经理在草稿上填写实际工程量和验收结论

**(3) 结论时效**
- 无强制时效,项目经理可随时出结论
- 系统每周催办一次(站内消息),提醒有未决验收单
- 不强制升级,不视同通过

**(4) 整改上限**
- 验收不通过 → 进入"整改中" → 整改完成后重新验收
- 整改次数上限 3 次,超出强制升级到管理层仲裁
- 仲裁结论:继续整改 / 终止该点位 / 特殊处理(需填写仲裁说明)

**(5) 验收单字段**
- 必填:实际工程量(与报价明细对比)
- 选填:客户确认附件(线下获取客户签字后上传)
- 系统自动:验收单编号、关联点位、关联报价、生成时间、项目经理

**(6) 验收与结算的关系**
- 验收通过 → 该点位报价金额进入结算队列
- 验收不通过 / 整改中 / 仲裁中 → 不可结算(系统强校验)

**(7) 系统使用范围约束** ⭐
- 本系统仅内部使用,客户不会登录系统
- 客户相关动作(确认、签字、付款承诺)均通过线下完成,内部人员上传附件留痕
- 系统不提供客户账号、客户端、客户门户

## 四、关键业务规则

| 规则编号 | 规则说明 |
|---------|---------|
| R1 | 维护型项目下必须至少有 1 个维保主合同(季度结算)和 0~N 个点位报价 |
| R2 | 维保主合同合同期为 3 个月整数倍且 ≥ 6 个月;采用合同季度(从生效日起算),每期 3 个月 |
| R2.1 | 季度结算单每期结束日次日自动生成草稿,状态流转:草稿→审核→确认→开票→回款→关闭 |
| R2.2 | 单期金额 = 合同总额 ÷ 期数;除不尽时尾差归入最后一期 |
| R2.3 | 合同金额变更经审批后从下一期生效,当期不补差 |
| R2.4 | 提前终止不自动结算,由商务线下协商后手动调整结算单(需财务审批 + 留痕) |
| R3 | 点位报价费必须验收通过后方可发起结算,系统强校验;整改中/仲裁中/验收不通过均不可结算 |
| R3.1 | 验收由项目经理单签生效;整改上限 3 次,超出升级管理层仲裁 |
| R3.2 | 系统仅内部使用,客户不登录;客户确认通过线下获取 + 内部上传附件留痕 |
| R4 | 同一点位支持多版本报价,但同一时间只有一个生效报价 |
| R5 | 两类费用(维保费/报价费)的结算单、发票、回款独立管理,互不混淆 |
| R6 | 点位验收不通过进入整改,整改期间不计算结算 |
| R7 | 维保主合同到期后,未结算的点位报价费仍可继续走验收结算流程 |

## 五、模块衔接关系(架构分层)

> 详细规则见 [模块五架构分层设计规格](docs/superpowers/specs/2026-06-26-模块五架构分层-design.md)

### 5.1 架构分层

```
┌─────────────────────────────────────────┐
│ 业务编排层:维护型项目(模块五)           │
│  - 编排点位报价、进度、验收、结算流程    │
│  - 不实现具体能力,调用基础能力层         │
└──────────────┬──────────────────────────┘
               │ 调用
┌──────────────▼──────────────────────────┐
│ 基础能力层:报价 / 进度 / 验收 / 结算     │
│  - 独立的能力服务,不感知维护型项目       │
│  - 通过 business_type 字段区分场景       │
└─────────────────────────────────────────┘
```

### 5.2 business_type 字段

| business_type | 含义 | 来源模块 |
|---------------|------|---------|
| NEW_BUILD | 新建工程报价/进度 | 模块二/三 |
| MAINTENANCE_POINT | 维护点位报价/进度 | 模块五 |

### 5.3 模块五职责边界

| 模块五做什么 | 模块五不做什么 |
|--------------|----------------|
| 维护型项目立项 | 报价 CRUD(调模块二) |
| 维护点位管理(CRUD) | 进度上报(调模块三) |
| 编排点位报价→施工→验收→结算流程 | 文件上传(调通用能力) |
| 双线结算看板(聚合查询) | 审批流(调通用审批) |
| 季度结算单生成(定时任务) | 验收单 CRUD(调模块三) |

### 5.4 模块衔接说明

| 模块 | 衔接关系 |
|------|---------|
| 合同管理 | 维护型项目主合同走合同管理;点位报价确认后可选择转为补充合同 |
| 报价上传存储 | 点位报价复用报价模块,通过 business_type=MAINTENANCE_POINT 标记 |
| 项目进度管理 | 维护型项目进度按点位维度管理,复用进度上报机制 |
| 项目维保 | 维护费季度结算与维保模块的维保合同/巡检任务打通 |

## 六、数据模型建议(关键实体)

### 6.1 项目基类与子类

```
Project(项目基类)
  ├─ id, code, name, customer_id, manager_id, address
  ├─ start_date, end_date, status
  ├─ type: NEW_BUILD / MAINTENANCE  ← 项目类型
  │
  ├─ 1:N → Equipment(设备台账) ← 设备归属项目
  │
  ├─[NEW_BUILD]→ 直接关联 Quote / Progress / Acceptance
  │
  └─[MAINTENANCE]→ MaintenanceProject(维护型项目扩展)
                       │
                       ├─ 1:N → MaintenancePoint(维护点位)
                       │          ├─ 1:N → Quote(business_type=MAINTENANCE_POINT, business_id=point_id)
                       │          ├─ 1:N → Progress(business_type=MAINTENANCE_POINT, business_id=point_id)
                       │          ├─ 1:1 → Acceptance(business_type=MAINTENANCE_POINT, business_id=point_id)
                       │          └─ 1:N → Equipment(设备,关联到具体点位)
                       │
                       ├─ 1:1 → MaintenanceContract(维保主合同)
                       │          └─ 1:N → QuarterlySettlement(季度结算单)
                       │          └─ 1:N → Equipment(合同覆盖设备清单)
                       │
                       └─ 1:N → PointSettlement(点位结算单, 验收后生成)
```

### 6.2 设备台账实体

```
Equipment(设备台账)
  ├─ id, code, name, brand, model, serial_number
  ├─ category(分类:门禁/监控/网络/楼控等)
  ├─ specs(规格参数), commissioning_date(投运日期)
  ├─ warranty_expiry(保修到期日), status(正常/故障/维修中/报废)
  ├─ project_id(归属项目), point_id(归属点位,维护型项目可填)
  └─ 1:N → MaintenanceRecord(维保记录)
```

### 6.3 基础能力层实体(通过 business_type 区分)

| 实体 | business_type 取值 | 说明 |
|------|-------------------|------|
| Quote(报价) | NEW_BUILD / MAINTENANCE_POINT | 通用报价表,通过字段区分新建工程报价与点位报价 |
| Progress(进度) | NEW_BUILD / MAINTENANCE_POINT | 通用进度表 |
| Acceptance(验收单) | NEW_BUILD / MAINTENANCE_POINT | 通用验收表 |

### 6.4 关键关系说明

- 基础能力层实体(Quote/Progress/Acceptance)通过 `business_type + business_id` 反向关联到业务实体
- NEW_BUILD 类型:business_id = project_id
- MAINTENANCE_POINT 类型:business_id = point_id
- 模块五(维护型项目)通过编排基础能力层实现业务流程,不重复定义报价/进度/验收实体

## 十、技术选型建议

> UI 风格定位:Linear/Notion 风(现代极简、克制配色、细腻阴影、紧凑高效)
> 详细设计规范见 [UI 设计规范](docs/superpowers/specs/2026-06-26-UI设计规范-design.md)

### 10.1 核心技术栈

| 层级 | 推荐技术 | 理由 |
|------|---------|------|
| 后端 | Java + Spring Boot | 企业级首选,强类型适合复杂业务,生态成熟,合同财务场景稳定性要求高 |
| 前端 | Vue 3 + Element Plus + 重写主题 | 生态成熟,通过 SCSS 变量覆盖调出 Linear/Notion 风格 |
| 数据库 | MySQL 8.x | 关系型数据,事务支持好,合同/结算数据强一致性 |
| 文件存储 | MinIO / 本地存储 | 合同、报价、验收资料文件存储,兼容 S3 协议 |
| 文件预览 | PDF.js + kkFileView | PDF/图片纯前端预览,Office 文档服务端转换,覆盖合同/报价/验收/维保/设备资料场景 |
| 图表 | Chart.js | 轻量,默认风格现代,符合 Linear 风 |
| 图标 | Lucide Icons | 线条简洁,符合 Linear 风 |
| 部署 | Docker + Nginx | 标准化部署,便于后续扩展 |

### 10.2 UI 设计规范要点

- **主色**: 靛蓝 `#4F6BED`
- **配色**: 纯白主区 `#FFFFFF` / 浅灰侧栏 `#FAFAFA` / 深灰文字 `#1F2937`
- **字体**: 中文 PingFang SC / 英文 Inter,正文 14px
- **圆角**: 按钮 6px / 卡片 8px / 弹窗 12px
- **阴影**: 极轻,`0 1px 2px rgba(0,0,0,0.04)` 默认,悬浮 `0 4px 12px rgba(0,0,0,0.08)`
- **间距**: 4 的倍数(4/8/12/16/24/32)
- **密度**: 舒适密度,表格行高 56px,表单间距 20px
- **动效**: 轻微动效(150-200ms ease),弹窗淡入+上移,hover 渐变
- **暗色模式**: 初期只做亮色,预留 CSS 变量
- **表格交互**: 查看为主,点击行打开右侧详情抽屉(非行内编辑)

# 模块五架构分层 设计规格

- **版本**: v1
- **日期**: 2026-06-26
- **状态**: 已确认,待落地
- **关联文档**: [需求文档.md](../../../需求文档.md) 第五节、第六节

---

## 一、背景

模块五(维护型项目管理)的"点位报价/点位进度/点位验收"与基础模块(报价/进度/维保)高度重叠。若不明确架构分层,会导致代码重复、边界不清、长期维护成本高。本规格定义"基础能力层 + 业务编排层"的架构方案,确保复用性与解耦性。

## 二、核心决策

- **架构**: 方案 A — 基础能力层 + 业务编排层
- **原则**: 报价/进度/验收作为基础能力,模块五编排调用,不重复实现
- **区分机制**: 基础能力表通过 `business_type` 字段区分业务来源

---

## 三、架构分层

```
┌─────────────────────────────────────────┐
│ 业务编排层:维护型项目(模块五)           │
│  - 编排点位报价、进度、验收、结算流程    │
│  - 不实现具体能力,调用基础能力层         │
└──────────────┬──────────────────────────┘
               │ 调用
┌──────────────▼──────────────────────────┐
│ 基础能力层:报价 / 进度 / 验收 / 结算     │
│  - 独立的能力服务,不感知维护型项目       │
│  - 通过 business_type 字段区分场景       │
└─────────────────────────────────────────┘
```

---

## 四、business_type 字段设计

| business_type | 含义 | 来源模块 |
|---------------|------|---------|
| NEW_BUILD | 新建工程报价/进度 | 模块二/三 |
| MAINTENANCE_POINT | 维护点位报价/进度 | 模块五 |

每种业务还可有 `business_id` 字段关联到具体业务实体(如 point_id)。

---

## 五、模块五职责边界

| 模块五做什么 | 模块五不做什么 |
|--------------|----------------|
| 维护型项目立项 | 报价 CRUD(调模块二) |
| 维护点位管理(CRUD) | 进度上报(调模块三) |
| 编排点位报价→施工→验收→结算流程 | 文件上传(调通用能力) |
| 双线结算看板(聚合查询) | 审批流(调通用审批) |
| 季度结算单生成(定时任务) | 验收单 CRUD(调模块三) |

---

## 六、数据模型关系

```
Project(项目基类)
  ├─ type: NEW_BUILD / MAINTENANCE
  │
  ├─[NEW_BUILD]→ 直接关联 Quote / Progress / Acceptance
  │
  └─[MAINTENANCE]→ MaintenancePoint(点位)
                        │
                        ├─→ Quote(business_type=MAINTENANCE_POINT, business_id=point_id)
                        ├─→ Progress(business_type=MAINTENANCE_POINT, business_id=point_id)
                        └─→ Acceptance(business_type=MAINTENANCE_POINT, business_id=point_id)
```

---

## 七、接口设计原则

基础能力层接口设计遵循"不知道上层"原则:

- `QuoteService.create(quoteDTO)` — 通用报价创建,不感知点位
- `MaintenancePointService.createPointQuote(pointId, quoteDTO)` — 模块五编排,内部调 `QuoteService.create` 并设置 business_type

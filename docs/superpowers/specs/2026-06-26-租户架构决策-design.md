# 租户架构决策 设计规格

- **版本**: v1
- **日期**: 2026-06-26
- **状态**: 已确认,待落地
- **关联文档**: [需求文档.md](../../../需求文档.md)

---

## 一、背景

系统使用范围为单公司内部,需明确租户架构以指导表结构设计和部署方案。多租户改造若初期未预留,后期改造成本极高,因此需在开发前确定。

## 二、核心决策

- **架构**: 单租户
- **理由**: 仅一家公司内部使用,无子公司隔离需求,无 SaaS 商业化计划

---

## 三、架构约束

| 项 | 决策 |
|----|------|
| 表结构 | 不加 tenant_id,所有业务表共用 |
| 数据隔离 | 通过 RBAC + 数据权限(本人/本部门/全部)实现 |
| 部署 | 单实例部署,单数据库 |
| 用户体系 | 全公司统一用户体系,按部门/角色区分权限 |
| 多公司扩展 | 不支持,如未来需要则按"新部署一套"方式实现 |

---

## 四、风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 未来要做 SaaS 改造成本高 | 接受此风险;真要做时通过抽象数据访问层 + 加 tenant_id 改造,工作量约 2-4 周 |
| 多部门数据权限需求 | 通过 RBAC + 数据权限字段(dept_id)实现,不需多租户 |

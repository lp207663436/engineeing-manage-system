🔴 跨层关键问题（最高优先级）

以下问题横跨前端、后端、数据库或文档多层，影响面广，需优先统一修复。
严重 跨层 数据权限范围定义三方不一致，导致越权或漏权

影响层：后端代码 + 前端展示 + 项目文档

后端定义（一致）：DataScopeHelper.java / DataScope.java / schema.sql 均为 1=全部, 2=本部门, 3=本部门及以下, 4=本人

前端表格展示映射（错误）：role/index.vue L94 dataScopeMap = { 1:'全部', 2:'本部门', 3:'本人', 4:'本部门及下属' } — 3 与 4 互换

前端编辑下拉（正确）：同文件 L155-158 1:全部数据, 2:本部门, 3:本部门及下属, 4:本人 — 与后端一致

项目文档（完全相反）：project_memory.md 记录 1本人/2本部门/3本部门及下属/4全部

后果：管理员在列表看到 dataScope=3 显示"本人"，编辑时下拉却选中"本部门及下属"，配置结果与预期相反，造成越权或漏权。

修复建议：统一前端 dataScopeMap 为 {1:'全部',2:'本部门',3:'本部门及下属',4:'本人'}；更新 project_memory.md 使其与代码一致。
严重 跨层 水平越权漏洞——所有业务模块 get/update/delete 无数据权限校验

影响层：后端所有业务 Service + 前端路由权限

后端：数据权限（@DataScope + DataScopeHelper.applyTo）仅在分页列表查询时生效。get(id)、update(dto)、delete(id) 直接 selectById(id) 后操作，不校验当前用户是否有权访问该数据。

前端：路由守卫中 meta.permission 校验逻辑已实现，但所有静态路由均未配置 meta.permission，导致该分支永不触发。已登录用户可通过 URL 直接访问无权页面。

后果：用户 A（data_scope=本人）在列表中看不到用户 B 的数据，但若知道 ID，可通过 GET/PUT/DELETE /{id} 直接访问/修改/删除他人数据（IDOR 漏洞）。

修复建议：①后端所有 get/update/delete 方法增加数据权限校验（封装 checkOwnership(entity)）；②前端为每条路由补充 meta.permission 或改为后端菜单动态生成路由。
严重 跨层 Logout 接口失效——Token 无法主动撤销

影响层：后端拦截器配置 + 前端登出流程

WebMvcConfig.java L26 将 /auth/logout 排除在 JwtInterceptor 之外，导致登出时 SecurityContext 未设置（ThreadLocal 为 null）。AuthController.logout() 中 SecurityContext.getUserId() 返回 null，authService.logout(userId) 永远不执行，Redis 中的 token 不会被删除。

后果：被盗 token 在 24 小时过期前无法通过登出撤销，安全风险高。前端虽然清理了本地 token，但旧 token 仍可被攻击者使用。

修复建议：将 /auth/logout 从 JwtInterceptor 排除路径中移除，使其需要经过 JWT 校验；或改为前端在 logout 请求中携带 token，后端手动解析后删除 Redis 缓存。
严重 技术 审批流金额阈值路由逻辑反转

文件：ApprovalService.java L84-96（startApproval）vs L185-194（approve）

startApproval 选择首节点条件：threshold == null || threshold >= businessAmount

approve 选择下一节点条件：threshold == null || businessAmount >= threshold

按 schema 注释"合同金额 ≥ 阈值才走该节点"，正确条件应为 businessAmount >= threshold（即 approve 的逻辑）。startApproval 的条件是反转的。

后果示例：节点 Node1(阈值5000)、Node2(阈值50000)，合同金额 20000：正确应走 Node1，实际代码从 Node2 开始审批——大金额合同反而跳过低阈值审批节点。

修复建议：将 startApproval L88 条件改为 node.getAmountThreshold().compareTo(businessAmount) <= 0（即 threshold ≤ amount）；兜底逻辑取 nodes.get(0) 而非 nodes.get(nodes.size()-1)。
严重 业务 验收通过自动建结算单触发 NOT NULL 违约（必现运行时异常）

文件：AcceptanceService.java L125-157（createPointSettlementDraft）

验收通过时自动创建点位结算单草稿，但插入 PointSettlement 时：

①未设置 code，而 point_settlement.code 为 NOT NULL，插入必失败；

②point_id 仅在 businessType=MAINTENANCE_POINT 时设置，而 point_id 为 NOT NULL，NEW_BUILD 类型验收通过会违约；

③amount 置为 BigDecimal.ZERO 注释"后续回填"，但无任何回填逻辑，违反规格"结算金额=报价金额"；

④未按 businessType 守卫，对 NEW_BUILD 验收也触发点位结算（点位结算仅适用于维护型项目）。

后果：验收通过操作整体抛 SQLException 失败，核心业务流程不可用。

修复建议：仅对 MAINTENANCE_POINT 触发；生成 code（如 PS + yyyyMMdd + 序号）；amount 直接从 Quote 取；校验 quoteId/pointId 非空。

业务合理性问题

严重业务 季度结算单生成时机与规格不符

文件：QuarterlySettlementService.java L127-185 / QuarterlySettlementJob.java L27-42

规格要求"每期结束日的次日 00:00 自动生成草稿单"，但 generateForContract 一次性遍历 periodCount 生成所有期次的草稿单。每日 00:00 跑的 Job 在合同创建后首次执行即把未来所有期次全部预生成。且单据内容缺"本期巡检次数、本期维保工单汇总、关联维保记录"字段。

修复建议：Job 改为只生成"昨日为某期结束日"的那一期；补充巡检次数/工单汇总/维保记录字段与聚合逻辑。
严重业务 维保主合同金额变更未联动季度结算

文件：ContractChangeService.java L88-125 / MaintenanceContractService.java

规则5要求"合同金额变更经审批后从下一期生效，当期不补差，结算单显示适用金额版本"。但 MaintenanceContract 无变更机制；已生成的未来期次结算单金额不更新；amountVersion 仅在手动 adjust 时自增，合同变更审批通过时未递增。

修复建议：为 MaintenanceContract 增加变更流程；审批通过后从下一期起按新金额重算并递增 amountVersion。

代码逻辑与安全问题

严重技术 JWT 密钥硬编码为默认值

文件：application.yml L27

secret: ${EMS_JWT_SECRET:ems-secret-key-must-be-at-least-256-bits...} 提供了公开的默认密钥。攻击者知道默认密钥后可伪造任意用户 token（包括超级管理员），完全绕过认证。

修复建议：移除默认值，改为 secret: ${EMS_JWT_SECRET}，启动时必须配置。
严重技术 数据库密码硬编码为默认值

文件：application-dev.yml L5

password: ${DB_PASSWORD:Abc123++} 提供了默认密码。未设置环境变量时使用弱密码连接数据库，且凭据可能泄露到代码仓库。

修复建议：移除默认值，改为 password: ${DB_PASSWORD}。
严重技术 Redis 无密码认证 + 数据库 useSSL=false

文件：application-dev.yml L8-17 / L3

Redis 连接未设置 password，若端口暴露可直接连接获取有效 token。MySQL 使用 useSSL=false + allowPublicKeyRetrieval=true，存在中间人攻击风险。

严重技术 无登录防暴力破解机制

文件：AuthService.java L32-48

登录接口无验证码、无失败次数限制、无账号锁定机制。攻击者可对 /auth/login 发起无限次暴力破解。

修复建议：增加 Redis 计数器实现登录失败次数限制（5 次失败锁定 30 分钟）。
严重技术 sys_role.data_scope 默认值为 1（全部数据）

文件：schema.sql L39

data_scope TINYINT DEFAULT 1（1=全部数据）。新建角色时若未显式指定 data_scope，默认拥有全部数据访问权限，违反最小权限原则。

修复建议：DDL 改为 DEFAULT 4（本人）。
警告技术 无 Token 刷新机制

JWT 有效期固定 24 小时，无 refresh token 机制。用户每 24 小时必须重新登录。

警告技术 BusinessException 未设置 HTTP 状态码

文件：GlobalExceptionHandler.java L19-22

所有 BusinessException（含 401/403）都返回 HTTP 200，仅在响应体 code 字段区分。监控系统无法识别异常。

警告技术 用户管理无自我保护

文件：SysUserService.java L60-79

update 不校验是否修改自己的状态/角色（可自我提权或自禁用）；delete 仅阻止删除超管，未阻止删除自己。

警告技术 角色管理未保护超管角色

文件：SysRoleService.java L53-64

update 未校验是否修改超管角色（code="admin"）。管理员可将超管角色 data_scope 改为本人，或修改 code 导致超管判断失效。

警告技术 菜单删除未检查子菜单

文件：SysMenuService.java L86-88

delete(id) 直接删除菜单，不检查是否有子菜单。删除父菜单后子菜单成为孤儿节点，菜单树构建异常。

警告技术 定时任务无并发执行保护

文件：NotificationScheduler.java / QuarterlySettlementJob.java / MaintenanceInspectScheduler.java

三个 @Scheduled 定时任务均未配置并发保护。多实例部署或任务执行跨触发周期时可能重复执行。

警告技术 AttachmentService.listByBusiness 未应用数据权限

文件：AttachmentService.java L33-39

listByBusiness 查询某业务实体所有附件，未调用 DataScopeHelper.applyTo。用户可查看任意业务实体的附件列表，存在越权风险。

警告技术 JwtInterceptor 每次请求查库性能问题

文件：JwtInterceptor.java L52/L80-84

每次 HTTP 请求都执行 selectById 查询用户、selectList 查询角色、selectBatchIds 查询角色信息。高频请求下数据库压力大。

修复建议：将用户基本信息缓存到 Redis（随 token 一起设置 TTL）。
警告技术 DbInitRunner 异常被静默吞没

文件：DbInitRunner.java 多处

多处 catch (Exception ignore) {} 静默吞没所有异常，可能掩盖真正的 SQL 错误。

建议技术 无密码复杂度校验

SysUserDTO 无 @Size/@Pattern 注解，创建用户时无最小长度、复杂度要求。

建议技术 无 XSS 防护处理

用户输入未做 HTML 转义/过滤直接存入数据库。前后端分离架构下风险较低，但后端无防御措施。

建议技术 SysDict/SysDictItem 无软删除和审计字段

字典表不继承 BaseEntity，无 deleted/create_by 等字段，删除为物理删除，与其他实体不一致。

代码逻辑问题

警告技术 路由级权限校验形同虚设

文件：router/index.ts L296-305

守卫实现了 meta.permission 校验逻辑，但所有静态路由均未配置 meta.permission，导致该分支永不触发。已登录用户可通过 URL 直接访问无权页面。

警告技术 响应拦截器对二进制流无兜底

文件：request.ts L30-31

拦截器强制解构 {code,message,data}。对 responseType:'blob' 等二进制响应无兜底，一旦有接口通过 service 返回二进制流，会因 code 为 undefined 而误报"请求失败"。

警告技术 TypeScript 类型安全薄弱，any 泛滥

文件：全局 36 个文件共 162 处 any

响应数据普遍用 any 解构，丢失类型推断与编译期校验。roleApi.update(... as any) 等绕过类型检查掩盖了真实错误。

修复建议：为每个 API 定义 PageResult<T> 返回类型，业务页用具体 interface 替换 any。
警告技术 字典 API 前缀与系统模块不一致

文件：api/system.ts L109-118

除字典外所有系统模块统一用 /sys 前缀，唯字典用 /system/dict。若后端 @RequestMapping 为 /sys/dict，则字典所有接口 404。

建议技术 403 未单独处理 / 登录页前缀图标可能不渲染 / 大量 pageSize:999 一次性拉取

request.ts 仅处理 401，403 落入通用错误提示未跳转 403 页；登录页 prefix-icon 传字符串但图标未全局注册可能不显示；多处下拉一次性拉取大页（999/1000），数据量增长后性能下降。

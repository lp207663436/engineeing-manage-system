Engine Manage System 全量代码审查报告

审查日期: 2026-06-28 ｜ 审查范围: 后端业务模块 + 后端基础设施/安全 + 前端全部代码 ｜ 技术栈: Spring Boot 3.2 + Vue 3.4 + Element Plus

整体评定: 需修复后可上线

本项目业务功能覆盖面广（22个Controller、20个Service、22个前端业务页面），架构分层基本合理，CRUD 基础完整。但在数据权限、安全防护、业务流程闭环、并发控制、前端权限控制五个核心维度存在系统性缺陷，共发现 105 个问题，其中严重问题 34 个需在上线前修复。

核心结论: 数据权限体系因 createBy 未自动填充而近乎失效；安全配置（JWT密钥、DB密码、CORS）不达生产标准；验收→结算、合同变更、收付款逾期等关键业务流程未闭环；前端动态菜单与路由权限未实现。

一、系统性问题主题（跨模块）

主题 1: 数据权限体系失效（影响面: 全系统）

  • 严重MetaObjectHandler 未填充 createBy，导致 dataScope=4（本人）时 create_by = null 永为 false，查不到任何数据
  • 严重EquipmentService.page 标注 @DataScope 但未调用 DataScopeHelper.applyTo()，注解形同虚设
  • 严重JwtInterceptor 无角色用户返回 dataScope=3（本部门及以下），注释说"仅本人"，数据范围被放大
  • 严重SettlementDashboardService / WorkbenchController / ContractPaymentService.dashboard 不校验数据归属，任何用户可查任意项目/合同数据
  • 中等DataScopeHelper「本部门及以下」只查两级部门，三层以上部门树数据遗漏

主题 2: 安全配置不达生产标准

  • 严重JWT 密钥硬编码在 application.yml 明文中
  • 严重数据库 root 密码 "Abc123++" 明文存储，Redis 无密码无连接池
  • 严重CORS allowedOriginPatterns("*") + allowCredentials(true) 完全开放
  • 严重/uploads/** 静态资源映射绕过 PreviewController 权限校验，用户可直接下载他人附件
  • 严重新用户默认密码 "123456" 硬编码，无密码复杂度校验
  • 严重GlobalExceptionHandler 将异常 message 直接返回客户端，泄露 SQL/类名等内部信息
  • 中等登录页硬编码 admin/admin123 默认凭据
  • 中等token 存 localStorage，存在 XSS 窃取风险

主题 3: 业务流程未闭环

  • 严重验收通过（result=PASS）不自动创建 PointSettlement，验收→结算链路断裂
  • 严重合同变更仅 AMOUNT 类型回写合同主表，START_DATE/END_DATE 变更审核通过后不生效
  • 严重ContractPayment 状态 OVERDUE 从未被设置，无定时任务将超期 PENDING 更新为 OVERDUE
  • 严重PointSettlement 状态流不完整，CONFIRMED/INVOICED 从未被设置，Dashboard"已结算金额"永远为 0
  • 严重审批流金额阈值（amountThreshold）未实现，标注 TODO
  • 严重NotificationScheduler 仅处理 ACCEPTANCE/WARRANTY，缺 SETTLEMENT/APPROVAL/OVERDUE 三类通知

主题 4: 前端权限控制缺失

  • 严重用户信息未持久化，刷新后 userId/username/name 全部丢失
  • 严重侧边栏菜单完全硬编码，未使用后端动态菜单，权限控制失效
  • 严重路由守卫仅校验 token，不校验路由访问权限，所有登录用户可访问全部页面
  • 严重userId 类型为 number 而非 string，Long ID 精度丢失

主题 5: 并发与数据一致性

  • 严重PointSettlementService.receive 先读后写，并发收款登记丢失更新（Lost Update）
  • 严重QuarterlySettlement 和巡检工单生成幂等检查采用"先SELECT再INSERT"非原子模式
  • 严重MyBatisPlusConfig 缺少乐观锁插件，quarterly_settlement.amount_version 无法生效
  • 中等ContractChangeService.update 无乐观锁，并发审核可能覆盖

三、必须修复的严重问题（34项）

3.1 数据权限体系失效（5项）

C1. MetaObjectHandler 未填充 createBy，数据权限本人范围完全失效
必须修复
模块: 后端基础设施 文件: config/MyBatisPlusConfig.java:22-35

问题: BaseEntity 的 createBy 标注了 @TableField(fill = FieldFill.INSERT)，但 MetaObjectHandler 的 insertFill 只填充 createTime 和 updateTime，未填充 createBy。导致 createBy 永远为 null，数据权限拦截器在 dataScope=4（本人）时 create_by = null 永为 false，查不到任何数据。

public void insertFill(MetaObject metaObject) {
    this.strictInsertFill(metaObject, "createTime", LocalDateTime.class, LocalDateTime.now());
    this.strictInsertFill(metaObject, "updateTime", LocalDateTime.class, LocalDateTime.now());
    // 缺少: this.strictInsertFill(metaObject, "createBy", Long.class, SecurityContext.getUserId());
}
修复: 在 insertFill 中增加 createBy 填充，并在 BaseEntity 增加 updateBy 字段在 updateFill 中填充。部分 Service（如 EquipmentService.create）手动设置了 createBy，但并非全部，应在框架层统一处理。
C2. EquipmentService.page 数据权限注解形同虚设
必须修复
模块: 后端业务 文件: service/EquipmentService.java:25-36

问题: EquipmentController.page 标注了 @DataScope，但 Service 层未调用 DataScopeHelper.applyTo(wrapper)，数据权限注解完全失效，任何用户都能查看全部设备数据。

修复: 在 wrapper.orderByDesc 之前添加 DataScopeHelper.applyTo(wrapper);，与 ContractService、ProjectService 等保持一致。
C3. JwtInterceptor 无角色用户数据范围返回值与注释矛盾
必须修复
模块: 后端安全 文件: security/JwtInterceptor.java:74-81

问题: 注释写"无角色默认仅本人"（应为4），但代码 if (userRoles.isEmpty()) return 3; 返回3（本部门及以下）。无角色用户反而能看到本部门及下属部门所有数据，数据范围被放大。

修复: 将 return 3 改为 return 4，与注释一致。
C4. Dashboard/Workbench 不校验数据归属
必须修复
模块: 后端业务 文件: controller/WorkbenchController.java:36-39、service/SettlementDashboardService.java、service/ContractPaymentService.java:154

问题: WorkbenchController 直接注入 Mapper 跳过 Service 层，统计查询（pendingAcceptances、overdueTasks 等）未应用数据权限，任何用户可查看全系统统计。SettlementDashboardService 和 ContractPaymentService.dashboard 同样不校验项目/合同归属。

修复: 通过 Service 层访问数据；Workbench 统计应考虑数据权限过滤；Dashboard 接口添加项目/合同归属校验。
C5. DataScopeHelper「本部门及以下」只查两级部门
必须修复
模块: 后端基础设施 文件: common/datascope/DataScopeHelper.java:46-54

问题: dataScope=3（本部门及以下）只查询了「本部门 + 直接子部门」两级。如果部门树有三层或更深，孙级及以下部门的用户数据被遗漏。

OR create_by IN (SELECT id FROM sys_user WHERE dept_id IN 
    (SELECT id FROM sys_dept WHERE parent_id = {0}))  -- 只查直接子部门
修复: 使用递归 CTE 或预先查询所有子孙部门 ID 集合，WHERE dept_id IN (?, ?, ...)。

3.2 安全漏洞（9项）

C6. JWT 密钥硬编码在配置文件
必须修复
模块: 后端基础设施 文件: resources/application.yml:27

问题: JWT 签名密钥 ems-secret-key-must-be-at-least-256-bits-long-for-hs256-algorithm 以明文存储，且被 JwtUtilTest 引用。代码仓库泄露后攻击者可伪造任意用户 token。

修复: 通过环境变量 ${EMS_JWT_SECRET} 注入，配置文件不落盘明文。
C7. 数据库密码和 Redis 明文存储
必须修复
模块: 后端基础设施 文件: resources/application-dev.yml:5,9-10

问题: 数据库 root 密码 "Abc123++" 明文存储；Redis 无密码保护、无连接池配置；使用 root 账户连接数据库。

修复: 密码改为 ${DB_PASSWORD} 环境变量注入；Redis 配置密码和 lettuce 连接池参数；生产禁用 root 账户。
C8. CORS 完全开放 + 允许携带凭证
必须修复
模块: 后端基础设施 文件: config/WebMvcConfig.java:32-38

问题: .allowedOriginPatterns("*") + .allowCredentials(true)，等同于完全放开跨域限制，攻击者可在恶意页面中携带用户 token 发起请求。

修复: 限定 allowedOriginPatterns 为具体域名列表（如 http://localhost:5173），通过配置文件注入白名单。
C9. /uploads/** 静态资源绕过权限校验
必须修复
模块: 后端基础设施 文件: config/WebMvcConfig.java:42-45

问题: 上传文件通过 /uploads/** 直接暴露为静态资源，已登录用户只需知道文件路径（如 /uploads/2026/06/uuid.pdf）即可直接下载他人附件，绕过 PreviewController 的业务权限校验。

修复: 将上传文件存储到不可直接访问的目录，所有文件访问统一通过 PreviewController/Download 接口并做权限校验；或移除静态资源映射。
C10. 新用户默认密码 "123456" 硬编码
必须修复
模块: 后端系统 文件: service/SysUserService.java:52-53

问题: 新用户不传密码则使用弱密码 "123456"，且无密码复杂度要求（无长度/大小写/数字/特殊字符校验）。

修复: 移除默认密码，强制要求传入密码；增加密码复杂度校验；首次登录强制改密。
C11. GlobalExceptionHandler 兜底异常暴露内部信息
必须修复
模块: 后端基础设施 文件: common/exception/GlobalExceptionHandler.java:41-43

问题: return Result.error(500, "服务器内部错误: " + e.getMessage()) 将原始异常 message 返回客户端，可能暴露 SQL 语句、类名、堆栈路径等敏感信息。同时缺少 HttpRequestMethodNotSupportedException、MaxUploadSizeExceededException 等异常类型的针对性处理。

修复: 兜底异常只返回通用提示"服务器内部错误"，将 e.getMessage() 记录到日志中；补充缺失的异常处理方法。
C12. SysRoleService.delete 用超管用户 ID 校验角色 ID
必须修复
模块: 后端系统 文件: service/SysRoleService.java:69

问题: SecurityContext.SUPER_ADMIN_ID（=1L）是用户 ID，但此处传入的 id 是角色 ID。恰好 admin 用户 ID=1、admin 角色 ID 也=1，所以表面上能工作，但这是巧合。超管角色 ID 不为 1 时校验失效，可被删除导致系统权限体系崩溃。

修复: 定义 SUPER_ADMIN_ROLE_ID 常量或通过 role.code="admin" 判断，与用户 ID 常量分离。
C13. SysNotificationService.page 当 userId 为 null 时返回全部通知
必须修复
模块: 后端系统 文件: service/SysNotificationService.java:67-78

问题: 当 Controller 传入 userId 为 null 时不追加 userId 过滤条件，返回所有用户的通知，存在数据泄露风险。

修复: 当 userId 为 null 时抛出异常或返回空结果，不做全量查询。
C14. AuthService 配置项与 application.yml 不匹配
必须修复
模块: 后端系统 文件: service/AuthService.java:29 vs resources/application.yml:28

问题: AuthService 读取 ${jwt.expire-seconds:86400}，但 yml 配置的是 ems.jwt.expire: 86400，两者不匹配，AuthService 永远使用默认值，修改配置不会生效。

修复: 统一为 ${ems.jwt.expire}，保持所有引用一致。

3.3 审批流缺陷（2项）

C15. ApprovalController.approve 缺少 @Valid 且 result 字段无校验
必须修复
模块: 后端业务 文件: controller/ApprovalController.java:35、dto/ApprovalDTO.java:13、service/ApprovalService.java:109-156

问题: Controller 未加 @Valid；DTO 的 result 字段无校验；Service 中 result 为 null 或任意非 "REJECTED" 值时均按"通过"处理。攻击者可传 null 绕过审批意图，将本应拒绝的审批自动推进到下一节点。

if ("REJECTED".equals(result)) { ... return; }
// else: 当作 APPROVED 处理，创建下一节点
修复: 1. Controller 加 @Valid；2. ApprovalDTO.result 加 @NotBlank 并限制值为 APPROVED/REJECTED；3. Service 显式判断 "APPROVED".equals(result)，不匹配时抛 BusinessException。
C16. 审批流金额阈值（amountThreshold）未实现
必须修复
模块: 后端业务 文件: service/ApprovalService.java:73,158、entity/ApprovalNode.java:17

问题: ApprovalNode.amountThreshold 字段已定义，但 startApproval() 和 approve() 中均标注 TODO 未实现。项目硬约束要求"审批流需支持按业务类型配置流程节点及金额阈值"。

修复: 在 startApproval() 中根据业务金额选择起始节点（首个 amountThreshold >= 业务金额 的节点）；在 approve() 中根据金额决定是否跳过后续节点。

3.4 业务流程未闭环（6项）

C17. 验收通过未自动触发结算生成
必须修复
模块: 后端业务 文件: service/AcceptanceService.java:83-107、service/MaintenanceTaskService.java:146-155

问题: AcceptanceService.submitResult() 当 result=PASS 时不自动创建 PointSettlement；MaintenanceTaskService.accept() 仅改状态为 COMPLETED，不创建 Acceptance 记录也不触发结算。验收→结算链路断裂。存在两套验收机制（MaintenanceTask.accept/reject 与 Acceptance 实体）未集成。

修复: AcceptanceService.submitResult() 在 result=PASS 时自动创建 PointSettlement 草稿；或将 MaintenanceTask.accept() 与 Acceptance 流程集成。
C18. 合同变更仅 AMOUNT 类型回写，其他类型变更不生效
必须修复
模块: 后端业务 文件: service/ContractChangeService.java:82-108

问题: audit() 方法中 changeField 为 START_DATE/END_DATE 时，审核通过后不回写合同主表对应字段，变更"通过"但实际不生效。且校验顺序错误（先更新后校验 newAmount>0）。

修复: 校验前置到更新之前；实现 START_DATE/END_DATE 等其他 changeField 的回写逻辑。
C19. ContractPayment 状态枚举不一致，OVERDUE 从未被设置
必须修复
模块: 后端业务 文件: entity/ContractPayment.java:48、service/ContractPaymentService.java:111-116

问题: 实体注释状态为 PENDING/RECEIVED/OVERDUE，Service 实际使用 PENDING/PARTIAL/RECEIVED（PARTIAL 不在枚举中，OVERDUE 从未被设置）。没有定时任务将超期 PENDING 更新为 OVERDUE。违反项目硬约束"收付款状态需使用PENDING/RECEIVED/OVERDUE"。

修复: 统一状态枚举；增加定时任务扫描 planDate < today && status = PENDING 的记录并更新为 OVERDUE。
C20. PointSettlement 状态流不完整，CONFIRMED/INVOICED 从未被设置
必须修复
模块: 后端业务 文件: entity/PointSettlement.java:24、service/PointSettlementService.java、service/SettlementDashboardService.java:38

问题: 实体定义状态 PENDING/CONFIRMED/INVOICED/RECEIVED/CLOSED，但 Service 中只设 PENDING 和 RECEIVED/PARTIAL。CONFIRMED 和 INVOICED 从未被设置，导致 Dashboard "已结算金额"永远为 0。

修复: 补全 confirm() 和 invoice() 状态流转方法；修正 Dashboard 状态聚合逻辑；将 PARTIAL 加入枚举或移除。
C21. MaintenanceTaskService.complete 未复制 partsUsed 到维保记录
必须修复
模块: 后端业务 文件: service/MaintenanceTaskService.java:127-140

问题: 完工自动生成维保记录时，任务的 partsUsed（配件使用）字段未复制到记录中，违反项目硬约束"维保任务必须包含配件使用字段"。code 使用 System.currentTimeMillis() 高并发下可能冲突。

修复: 添加 record.setPartsUsed(existing.getPartsUsed())；code 改用 UUID 或雪花算法。
C22. NotificationScheduler 通知类型覆盖不全
必须修复
模块: 后端系统 文件: scheduler/NotificationScheduler.java

问题: 只处理了 ACCEPTANCE（验收催办）和 WARRANTY（合同到期）两种通知。SETTLEMENT（结算）、APPROVAL（审批）、OVERDUE（逾期）三种类型未在定时任务中处理。合同收付款逾期通知缺失。

修复: 增加 scanOverduePayments 方法扫描 contract_payment 表中 status=PENDING 且 plan_date < now() 的记录发送 OVERDUE 通知；补充 SETTLEMENT/APPROVAL 通知触发。

3.5 并发与数据一致性（4项）

C23. PointSettlement 收款登记并发丢失更新
必须修复
模块: 后端业务 文件: service/PointSettlementService.java:91-116

问题: receive() 先读取 receivedAmount，在 Java 中累加，再以绝对值写回。两个并发请求同时读取相同 currentReceived，各自加自增额写回，后写入者覆盖先写入者，导致收款金额丢失。

BigDecimal currentReceived = existing.getReceivedAmount();
BigDecimal totalReceived = currentReceived.add(receivedAmount);
// ...
.set(PointSettlement::getReceivedAmount, totalReceived)  // 用计算后的绝对值覆盖
修复: 使用 SQL 原子自增 wrapper.setSql("received_amount = received_amount + " + receivedAmount)，或对记录加乐观锁/悲观锁。
C24. 季度结算与巡检工单生成幂等检查非原子
必须修复
模块: 后端业务 文件: service/QuarterlySettlementService.java:147-155、job/MaintenanceInspectScheduler.java:51-57
